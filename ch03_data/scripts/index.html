    Chapter 3 Synthetic Data Scripts

    These scripts generate the fictional healthcare and business records used in Chapter 3 and later chapters. Run commands from the repository root.

    The folder roles are:

    • ch03_data/scripts/: commands readers run.
    • ch03_data/generation_modules/: import-only implementation modules.
    • ch03_data/output_data/generated_data/: synthetic datasets created by the generator.
    • ch03_data/output_data/public_reference/: downloaded or locally proxied public datasets.
    • ch03_data/output_data/analysis_results/: outputs from data-quality and analysis scripts.

    Setup

    From the project root, where pyproject.toml is located, run:

    uv sync

    This creates a shared .venv/ environment at the project root. Do not create separate environments inside ch03_data/ or ch03_data/scripts/.

    With standard Python:

    python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install pandas matplotlib

    On Windows, activate the environment with:

    .venv\Scripts\activate

    The generator uses only the Python standard library. Pandas and Matplotlib are required by the data-quality, phenotyping, and public-data helper scripts. In the main documented path below, use uv run python ... for all chapter commands unless you have already activated .venv/.

    Generate the Full Data Package

    With uv:

    uv run python ch03_data/scripts/generate_all_synthetic_data.py

    With an activated standard Python environment:

    python ch03_data/scripts/generate_all_synthetic_data.py

    The default run creates 20,000 fictional patients and writes the data to ch03_data/output_data/generated_data/.

    Command-Line Options

    --patients          Number of fictional patients. Default: 20000.
--seed              Random seed used for reproducibility. Default: 20260609.
--output-dir        Output directory. Default: ch03_data/output_data/generated_data/.
--diagnosis-dict    Optional diagnosis-code CSV. Built-in codes are used when omitted.
--drug-dict         Optional drug-code CSV. Built-in codes are used when omitted.

    Example:

    uv run python ch03_data/scripts/generate_all_synthetic_data.py \
  --patients 5000 \
  --seed 42 \
  --output-dir ch03_data/output_data/generated_data_demo

    Use the same seed and options to reproduce the same records.

    Supply custom code dictionaries

    Diagnosis and drug dictionaries are optional because they map generated events to clinical and product concepts that may come from external reference files. Other values, such as message topics and fictional company names, are scenario assumptions and remain defined in the modules.

    To use custom dictionaries:

    uv run python ch03_data/scripts/generate_all_synthetic_data.py \
  --diagnosis-dict path/to/diagnosis_codes.csv \
  --drug-dict path/to/ndc_codes.csv

    The diagnosis file must contain a code column named CODE, code, or Code. An optional description can be provided in CODE_DESCRIPTION, DESCRIPTION, or description. CODE_ORIG is also supported.

    The drug file must contain at least two usable rows and an NDC column named NDC, ndc, or Ndc. The first row is treated as the launch product, and the remaining rows are treated as comparators. Supported optional fields include BRAND_GENERIC, CUI_L1_NAME, BRAND, CUI_L2_NAME_ARRAY, INGREDIENT_NAME_ARRAY, and ALL_INGREDIENTS.

    The loader reads up to 40 usable rows from each file. An omitted option uses the built-in dictionary. An explicitly supplied file that is missing or does not meet these requirements stops the run with an error. The selected dictionaries are written to ch03_data/output_data/generated_data/reference/ so the generated events remain interpretable. The package manifest records whether each dictionary was built in or supplied from an external path, along with the run seed, requested patient count, schema version, row counts, date ranges, and table contracts for the generated files.

    Script Guide

    Script Purpose Direct command-line options

    generation_modules/scenario.py Creates the shared Roventra treatment-attempt path that links EHR, specialty-pharmacy, and pharmacy events No
    generation_modules/claims.py Creates medical and pharmacy claims No
    generation_modules/ehr.py Creates encounters, diagnoses, laboratory results, problem lists, and medication orders No
    generation_modules/crm.py Creates field interactions, sales-region alignment, and HCP digital engagement No
    generation_modules/specialty_pharmacy.py Creates specialty-pharmacy referrals, authorization decisions, and shipments No
    generation_modules/formulary.py Creates access rules, formulary status, and formulary history No
    generation_modules/open_payments.py Creates synthetic Open Payments-style records No
    generation_modules/cms_partd.py Creates synthetic Medicare Part D prescriber summaries No
    generation_modules/data_noise.py Adds controlled claim lag, duplicates, and missing product mappings No
    data_quality.py Runs data-quality checks against the generated records No
    build_main_figures.py Builds the main Chapter 3 manuscript figures from the current generated package No
    defect_impact_demo.py Quantifies each planted defect's before/after impact and writes supporting detail tables No
    lab_phenotyping.py Builds laboratory phenotypes and creates the A1C distribution notebook figure No
    plot_claim_lag.py Creates the medical-claim receipt-lag distribution chart for notebook or appendix use No
    build_conceptual_figures.py Legacy conceptual-figure builder kept for editable diagram experiments No

    The modules under ch03_data/generation_modules/ are called by generate_all_synthetic_data.py. They are Python functions, not separate command-line programs.

    Two executed companion notebooks live next to the chapter text in ch03_data/: chapter3_walkthrough.ipynb (the chapter as one executable story) and exercise_solutions.ipynb (worked answers to the chapter exercises). Re-execute them after regenerating the data so embedded outputs stay current.

    What Can Be Configured

    The supported command-line settings are patient count, random seed, output directory, and optional diagnosis and drug dictionaries. When the dictionary options are omitted, the generator uses built-in synthetic codes. Account and provider counts scale automatically with the patient count.

    Most domain assumptions are currently constants inside the modules. Examples include claims per patient, treatment probabilities, event dates, formulary change rates, duplicate rates, and missing product-mapping rates. Change these values in the relevant module when an exercise requires a different scenario:

    Setting Location
    Regions, specialties, channels, products, and entity scaling generation_modules/entities.py
    Claims per patient and prescription patterns generation_modules/claims.py
    Claim duplicate and missing-mapping rates generation_modules/data_noise.py
    Shared launch-treatment scenario, access friction, and fulfillment outcomes generation_modules/scenario.py
    EHR event and laboratory distributions generation_modules/ehr.py
    Field and digital interaction patterns generation_modules/crm.py
    Specialty-pharmacy pathway probabilities generation_modules/specialty_pharmacy.py
    Coverage rules and formulary-change rates generation_modules/formulary.py
    Open Payments assumptions generation_modules/open_payments.py
    Medicare Part D prescribing distributions generation_modules/cms_partd.py

    These internal assumptions are not command-line options. Keep changes small, document them, and use a fixed seed when comparing scenarios.

    Data-Quality Analysis

    Run:

    uv run python ch03_data/scripts/data_quality.py

    The script checks medical-claim duplicate flags and receipt lag, pharmacy product mappings, provider taxonomy consistency, and patient coverage for a prespecified observation window. It writes detailed CSV files and a headline summary under ch03_data/output_data/analysis_results/data_quality/.

    The main functions are medical_claim_duplicate_audit, claim_lag_analysis, product_mapping_audit, provider_taxonomy_audit, coverage_observation_audit, and run_data_quality. The specialty-to-taxonomy map is imported from generation_modules.entities so the audit can never drift from the generation rules.

    Defect Impact Demonstrations

    Run:

    uv run python ch03_data/scripts/defect_impact_demo.py

    The script runs before/after comparisons for the claim-maturity false decline and brand-count recovery from the NDC reference. It writes detail CSVs to ch03_data/output_data/analysis_results/defect_impact/.

    Public Reference Downloads

    Run the live API path with:

    uv run python ch03_data/scripts/download_public_data.py

    Run the offline contract-compatible path with:

    uv run python ch03_data/scripts/download_public_data.py --dry-run

    Both modes now write the same top-level schemas for cms_partd.csv and open_payments.csv, including provenance columns such as data_source, source_url, extract_year, source_mode, schema_version, and retrieved_at_utc. The prevalence anchor file nhanes_prevalence.csv also carries row-level source metadata.

    EHR Phenotyping Analysis

    Run:

    uv run python ch03_data/scripts/lab_phenotyping.py

    The script keeps the latest A1C, PSA, and PD-L1 result per patient, applies prespecified analytical bands, and joins the classes back to the full patient table. Patients without a result remain in the cohort with the class no_result. It writes the patient-level cohort, a summary CSV, and a1c_patient_group_distribution.png under ch03_data/output_data/analysis_results/lab_phenotyping/.

    The main functions are classify_by_a1c, classify_by_psa, classify_by_pdl1, build_lab_phenotype_cohort, lab_phenotype_summary, and run_phenotyping.

    The script also exports the publication figure as SVG and PNG under ch03_data/assets/figures/.

    Rebuild the Chapter Figures

    The main manuscript figures are rebuilt from the current generated package.

    uv run python ch03_data/scripts/build_main_figures.py

    The script writes SVG and PNG files under ch03_data/assets/figures/. Figures 3.1 and 3.3 read values from the generated records, so regenerate data before rebuilding figures when the seed, patient count, or generator assumptions change.

    The older scripts defect_impact_demo.py, plot_claim_lag.py, and lab_phenotyping.py still produce supporting analysis outputs used by the notebook and appendix material.
